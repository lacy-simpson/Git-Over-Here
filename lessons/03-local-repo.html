Set up Git on your machine, start a local repository, and learn the local workflow before GitHub enters the picture.

In this lesson, you will work entirely on your own machine: initialize a repository, inspect its state, stage a file, save a commit, and create a branch so you can see how Git manages work locally before anything is shared to GitHub.

Local Setup

1

Verify Git Is Installed

Open your terminal and confirm Git is available on your machine.

Terminal
git --version
# You should see: git version 2.x.x # Any 2.x version is fine
Git isn't installed — open for install steps
🍎 macOS
  1. Check for Homebrew: brew --version
  2. If needed, install Homebrew from brew.sh
  3. Install Git: brew install git
  4. Re-run git --version
🪟 Windows
  1. Go to git-scm.com/download/win
  2. Run the installer with the defaults
  3. Open Git Bash and run git --version
🐧 Linux
  1. Run sudo apt-get update && sudo apt-get install git
  2. Re-run git --version
2

Configure Your Identity

Git stamps your name and email on every commit. Use the same email as your GitHub account.

Terminal
git config --global user.name "Your Name" git config --global user.email "you@uipath.com" git config --list # verify user.name and user.email
✓
State Check — git config --list shows your name and email. Your local Git setup is ready.

Start a New Local Repository

Terminal
git init my-project cd my-project
# Git creates a hidden .git folder inside my-project # That folder is what turns it into a repository
✓
State Check — Run git status. You should see that Git is tracking this folder even though there are no commits yet.

Git Init vs Git Clone

📦
Use git init for a brand-new local project and use git clone when a repository already exists remotely and you want a copy on your machine. You will use the git clone pattern in the Boss Battle.

Stage and Commit Locally

1

Check Your State

Terminal
git status

Use git status before and after every action. It tells you exactly what Git sees.

2

Create a File to Track

Terminal
touch praise.txt echo "Diptendu is the best" > praise.txt cat praise.txt
# Diptendu is the best

Now Git has a real file to notice. If you run git status again, you should see praise.txt as a new untracked file.

3

Stage the File

Terminal
git add praise.txt

Staging tells Git: include this exact file in the next commit. In this example, you are staging praise.txt.

4

Commit the Snapshot

Terminal
git commit -m "Add praise.txt" git log --oneline # verify your commit appears
✓
State Check — Your new commit appears at the top of git log --oneline. At this point your work is saved locally, even with no internet connection.

Branching

Branches are part of the local Git workflow too. You can create, switch, and work in branches entirely on your own machine before anything ever goes to GitHub.

Without branches

Every experiment happens on the same line of history. Harder to isolate work, easier to lose track of what changed.

With branches

A safe workspace for one change, one idea, or one fix — without disturbing your main line of work.

Terminal
git checkout -b your-name/my-change # create + switch to a new branch git branch # list local branches git checkout main # switch back to main
# git branch output looks like: # main # * your-name/my-change

Convention Name branches your-name/description so the purpose is obvious to anyone scanning the repo later.

Key Takeaway

Your local repository is where Git starts. If you understand how to initialize, inspect, stage, commit, and branch locally, the remote workflow makes far more sense later.
