What This Course Is Not

This workshop is not intended to be a deep dive into the full breadth of Git. There are many excellent resources available if you want to build a stronger long-term foundation and go deeper into Git concepts, workflows, and edge cases.

This course is intentionally practical. The goal is to give you a quick overview of the basic functionality you need most, then help you use Git in a real application so the workflow feels useful and approachable right away.

Why This Course Matters

If you plan to work with Claude Code, Codex, or any other AI coding tool, Git is not optional background knowledge anymore. It is the skill that lets you safely review what the tool changed, save your progress, roll back mistakes, and share work with confidence instead of crossing your fingers.

For GPS, this becomes even more real. If you want to contribute to agents, skills, prompts, or other shared projects, Git is part of the job. It is how you show your work clearly, collaborate without stepping on someone else's changes, and participate in the same workflow the rest of the team will expect.

If you have ever thought, "I can probably get by without learning this part," this is the part that will eventually slow you down. The good news is that you do not need to master all of Git today. You just need enough of it to start working confidently in real situations, and that is exactly what this course is designed to help you do.

About Git

What Is Git?

Git is a version control system. It tracks changes over time, lets you compare versions, recover earlier work, and collaborate without overwriting teammates. Long before AI coding tools existed, Git was already the foundation of how software teams work safely.

Why Git With AI Coding Tools?

AI tools like Claude Code and Codex generate and modify code fast. Without version control you can't track what changed, roll back a mistake, or safely collaborate. Git makes AI-generated code auditable, reversible, and team-ready.

Key Terminology

  • 📁
    Repository (Repo)A project folder tracked by Git. Contains all files AND the complete history of every change — nothing is lost.
  • 📸
    CommitA labelled snapshot of staged changes. Think of it as a save point you can always return to.
  • 🌿
    BranchAn independent line of development. Work freely on a branch without affecting the main codebase until you're ready.
  • ☁️
    RemoteA hosted version of your repo (GitHub, GitLab). This is where you share and collaborate.

Local vs. Remote

Local RepoRemote Repo
Lives on your machineLives on GitHub
Your day-to-day workspaceWhere the team shares work
Works offlineRequires internet
Only you can see it until you pushAccessible to all collaborators

4 Stages & 3 Commands

This diagram explains the basic Git commands you'll use today
  1. 01

    Working Directory

    Files as you edit them. Git sees changes but isn't tracking them yet.

  2. git add Select which changes go into your next commit
  3. 02

    Staging Area

    Changes you've selected for the next commit. Your review queue before saving.

  4. git commit Save a labelled snapshot in your local history
  5. 03

    Local Repo

    Committed history saved in the hidden .git folder on your machine. Works offline.

  6. git push Send your commits to GitHub so teammates can see them
  7. 04

    Remote Repo

    The shared copy on GitHub. Everyone on the team pushes and pulls from here.

↺
git pull moves in the opposite direction — brings remote changes down to your local repo and working directory.

Stage and Commit Locally

1

Check Your State

Terminal
git status

Use git status before and after every action. It tells you exactly what Git sees.

2

Create a File to Track

Terminal
touch praise.txt echo "Diptendu is the best" > praise.txt cat praise.txt
# Diptendu is the best

Now Git has a real file to notice. If you run git status again, you should see praise.txt as a new untracked file.

3

Stage the File

Terminal
git add praise.txt

Staging tells Git: include this exact file in the next commit. In this example, you are staging praise.txt.

4

Commit the Snapshot

Terminal
git commit -m "Add praise.txt" git log --oneline # verify your commit appears
✓
State Check — Your new commit appears at the top of git log --oneline. At this point your work is saved locally, even with no internet connection.

Branching

Branches are part of the local Git workflow too. You can create, switch, and work in branches entirely on your own machine before anything ever goes to GitHub.

Without branches

Every experiment happens on the same line of history. Harder to isolate work, easier to lose track of what changed.

With branches

A safe workspace for one change, one idea, or one fix — without disturbing your main line of work.

Terminal
git checkout -b your-name/my-change # create + switch to a new branch git branch # list local branches git checkout main # switch back to main
# git branch output looks like: # main # * your-name/my-change

Convention Name branches your-name/description so the purpose is obvious to anyone scanning the repo later.

Key Takeaway

Your local repository is where Git starts. If you understand how to initialize, inspect, stage, commit, and branch locally, the remote workflow makes far more sense later.

04⏱ 20 min

Git - Remote Repo

Now take that local repository and connect it to GitHub so you can push your work up and pull changes down.

What We'll Cover
  • Create a new repository in GitHub
  • Connect your local repo to that remote
  • Push your local commits to GitHub
  • Pull and interpret the "up to date" result

Step 1 — Create the Remote Repo in GitHub

  1. Go to GitHub and click New repository
  2. Give the repo a clear name
  3. Set visibility appropriately for the workshop or project
  4. Leave README, .gitignore, and license unchecked because your local project already exists
  5. Create the repository and keep the page open
🌐
Pay attention to visibility when creating a new GitHubPublic repositories can be viewed by anyone on the internet. That means code, prompts, notes, commit history, and anything you push may be exposed beyond your team. If the work is not clearly safe to share, do not use a public repo.
💡
Why leave the repo empty?If GitHub adds starter files and your local project already has its own first commit, you create unnecessary extra history right away.

Step 2 — Connect Local to Remote

GitHub will show you commands for an existing repository. That is the path we want here.

Terminal
git remote add origin https://github.com/your-org/your-repo.git git branch -M main git remote -v
# origin https://github.com/your-org/your-repo.git (fetch) # origin https://github.com/your-org/your-repo.git (push)
✓
State Check — git remote -v shows origin for fetch and push. Your local repo now knows where GitHub lives.

Step 3 — Push Your Changes

Terminal
git push -u origin main

-u sets the upstream relationship, which means later you can usually just run git push without repeating the branch name.

Authenticate with GitHub if the push prompts you
Option A — Personal Access Token (PAT) — Recommended for this workshop
  1. Go to GitHub → Settings → Developer settings → Personal access tokens
  2. Generate a token with repo access
  3. When Git asks for a password during push, paste the token instead
Option B — SSH key Use SSH if you work in GitHub regularly and want to avoid repeated prompts.
✓
State Check — Refresh the GitHub repo page. You should now see your local files in the remote repository.

Step 4 — Pull the Latest Changes

Terminal
git pull origin main
# From https://github.com/your-org/your-repo # Already up to date.

That message is expected in this lesson. It means your local branch already matches the remote. Later, when teammates change the repo, this is how you bring their updates down.

Key Takeaway

A remote repository is just the shared copy of your local history. First you connect it with git remote add origin, then you publish with git push, and later you stay in sync with git pull.

Git Collaboration

This is where branches, pull requests, review, merge, and merge conflicts come together as a team workflow.

What We'll Cover

  • How to open a pull request from your branch
  • What happens when the PR is approved and merged

Why Collaboration Starts With Branches

Branches are still the foundation here. Instead of changing main directly, each person works in their own branch, then uses a pull request to propose those changes for review.

  • Work stays isolated until it is ready
  • Review happens before main changes
  • It is easier to understand who changed what
  • Bad changes are easier to stop or roll back

What Is a Pull Request?

A Pull Request (PR) is a formal request to merge your branch into another branch, usually main. It creates a space for review before the shared version changes.

  • Review code changes before they hit main
  • Leave comments and suggestions
  • Approve or request changes
  • Merge only after the change is understood

Opening a Pull Request

1

Push your branch

Terminal
git push origin your-branch-name
2

Open the repo on GitHub

GitHub often shows a banner for your recently pushed branch. If not, go to Pull requests → New pull request.

3

Set base and compare

Base is usually main. Compare is your branch.

4

Write a useful PR title and description

Your reviewer should understand what changed and why without opening every file.

5

Create the PR

Submit it for review and wait for comments, approval, or requested changes.

After the Pull Request

  • 1
    Reviewer leaves feedbackThey may approve, ask questions, or request changes.
  • 2
    You update the branch if neededMake changes, commit them, and push again. The PR updates automatically.
  • 3
    The PR gets mergedOnce approved, your branch is merged into main and becomes part of the shared version.

Merging the Branch

When the PR is approved, GitHub merges the branch into main. After that, pull the latest main locally so your machine matches the shared repo again.

Terminal
git checkout main git pull origin main
🔁
What learners should notice — the changes from the merged branch now appear on main. That is the full collaboration loop working end to end.

Merge Conflicts — What They Look Like

A merge conflict happens when two branches changed the same line in the same file. You are not expected to resolve one live today, but you should recognize the markers.

What Git shows in the file
<<<<<<< HEAD prompt: "Summarize top 3 case trends" ======= prompt: "Identify top 5 trends and root causes" >>>>>>> teammate/improve-skill-prompt

Your version is above the divider. The teammate version is below. Git stopped and needs a human decision.

How to resolve
  • 1
    Open the fileFind the conflict markers and decide what final version to keep.
  • 2
    Delete the markersKeep only the content you want in the final file.
  • 3
    Stage and commit againGit then records the resolved version.

Git Reference Sheet

Reference Page
Git Quick Reference

This section is designed to read like a clean internal reference page you can come back to later. Use it as your short list of commands, habits, and reminders when Git starts to feel noisy.

📝
Save this for laterCopy these notes into Confluence, Notion, or your own personal cheat sheet so you have one place to revisit the most useful Git habits from this workshop.

Everyday commands

  • git status tells you what Git knows right now. Use it before and after major actions.
  • git log --oneline lets you quickly scan recent commit history.
  • git branch shows which branch you are on.
  • git remote -v verifies where your repo pushes and pulls.
  • git restore --staged filename unstages a file.
  • git checkout main switches you back to main.
  • git pull origin main syncs the latest shared version.
  • git push publishes your current branch after upstream is set.

Commit message pattern

Write it so a teammate or future-you knows what changed without opening the file. Start with an action word such as Add, Fix, or Remove.

Good examples
  • Add Slack search step to skill
  • Fix broken nav on mobile
  • Update README with setup steps
  • Remove unused import in app.js
Avoid these
  • fix
  • stuff
  • asdfgh
  • updated file

Common mistakes to avoid

  • Never commit .env files, API keys, passwords, tokens, or client data.
  • Do not work directly on main when using a shared repository.
  • Do not use vague commit messages that hide what changed.
  • Do not push changes you did not mean to stage.
  • Do not skip review and merge without a pull request.

Daily habits that reduce Git stress

  • Run git status before and after major actions.
  • Work on a branch instead of directly on main.
  • Write small, focused commits.
  • Pull before starting shared work.

When you are stuck

  1. Run git status.
  2. Read exactly what Git is telling you.
  3. Confirm your branch with git branch.
  4. Pull if you need the latest shared changes.
  5. Ask for review before merging.
Key Takeaway

The most useful Git reference is the one you will actually reuse. Keep a short, practical cheat sheet nearby: status, branch, pull, push, focused commits, and no secrets.

Intro
What does "origin" mean?

origin is just a nickname — the default name Git gives to the remote repository you cloned from. It is not a special command or a fixed rule.

When you run git clone https://github.com/org/repo.git, Git automatically creates a remote named origin pointing to that URL. So when you later run git push origin main, you are saying: "push my commits to the remote called origin, onto the branch called main."

You can see all your remotes and their URLs by running git remote -v in your terminal.

What does cd do?

cd stands for "change directory." It moves your terminal into a folder — the same way double-clicking a folder in Finder or File Explorer opens it.

Common uses:

  • cd my-folder — move into a folder named "my-folder"
  • cd .. — move up one level (to the parent folder)
  • cd ~ — jump back to your home directory

To see where you currently are, run pwd (Mac/Linux) or type cd with no arguments (Windows).

Why does Git have a staging area?

Most save systems are all-or-nothing. Git's staging area gives you a "review queue" before you lock in a commit.

Real scenario: You edited three files today — two are part of one bug fix, and one is an unrelated experiment. Without staging, you'd have to commit all three together, making your history hard to read.

With staging, you git add the two related files, commit them as a clean, focused snapshot, then decide what to do with the third separately.

Think of it like putting items in a cart before checkout — you can review and remove before finalizing.

SSH Keys vs. Personal Access Tokens

Both methods authenticate you to GitHub — they're just different approaches with different tradeoffs.

Personal Access Token (PAT): A long string you paste as your "password" when Git asks. Easy to set up in 2 minutes. You may need to re-enter it depending on your credential settings.

SSH Key: A key pair generated on your machine. Once you add your public key to GitHub, Git never asks for credentials again. Takes ~5 minutes to set up but saves time long-term.

For this workshop: PAT is faster. For regular daily use: SSH is smoother.

git switch — the modern alternative

Modern Git (2.23+) introduced git switch as a cleaner, more focused alternative to git checkout for working with branches.

They are equivalent:

  • git checkout -b feature/name ≡ git switch -c feature/name
  • git checkout main ≡ git switch main

This guide uses git checkout because it works in all Git versions. If you see git switch in other tutorials, it's doing the same thing.

What is .gitignore?

A .gitignore file tells Git which files to never track — even if they exist in your project folder.

Common entries: .env files with secrets, node_modules/, compiled output, OS junk like .DS_Store.

Without it, git add . can accidentally stage things you never want in version control. It's one of the first files most developers add to a new project.

Working With a Local Repository

A local repository is the copy of your project that lives on your machine. This is where you initialize Git, inspect file changes, stage work, create commits, and create branches before anything is shared remotely.

That means a lot of Git happens without the internet. You can make commits, inspect history, and organize work locally first, then connect the repository to GitHub later when you are ready to push.

If the local workflow makes sense to you, the remote and collaboration workflow becomes much easier to understand.
